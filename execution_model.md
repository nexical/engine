# Signal-Based State Machine Orchestration and Agent Execution System

## Introduction

This document provides a comprehensive guide to the execution model used in this project. It is designed to educate contributors and users on how the system orchestrates multiple agents to complete complex tasks, how it bridges different tools (like CLI applications), and how it manages state to allow for resilient, resumable workflows.

The core of the system is a **Signal-Based State Machine Orchestrator**. Unlike linear script execution, this system is dynamic. It reacts to "signals" from agents—requests to change the plan or architecture based on new discoveries—allowing for adaptive and intelligent problem-solving.

## Core Concepts

### 1. Signal-Based Orchestration
In a traditional workflow, a controller dictates actions to agents. In this system, agents can "talk back" to the controller via **Signals**. A signal is a file written by an agent that indicates a need to deviate from the current path.
- **REPLAN**: "I can't do this task as described, or I found a better way. Please update the plan."
- **REARCHITECT**: "The fundamental approach is wrong. We need to rethink the entire solution."

### 2. The State Machine
The Orchestrator operates as a Finite State Machine (FSM) with the following states:
- **IDLE**: Waiting for input.
- **ARCHITECTING**: Designing the high-level solution.
- **PLANNING**: Breaking the solution into actionable tasks.
- **EXECUTING**: Running the tasks.
- **INTERRUPTED**: Paused due to a signal, ready to resume or pivot.
- **COMPLETED**: All tasks finished successfully.
- **FAILED**: Max loops reached or unrecoverable error.

### 3. Agent Execution Bridge
The system is designed to be agnostic to *how* an agent is implemented. It uses **Plugins** to bridge the Orchestrator to the actual agent code.
- **CLI Plugin**: The primary bridge. It runs agents as external CLI processes. This allows you to write agents in any language (Python, Rust, Node.js) as long as they can be called from the command line.
- **Persona System**: Agents can assume "Personas" (e.g., "Senior Backend Engineer") which inject specific context and behaviors into their prompts.

## System Architecture

The system is composed of four main components managed by the `Orchestrator` class.

### 1. The Orchestrator (`src/orchestrator.ts`)
The brain of the operation. It:
- Manages the `state.yml` file.
- Transitions between states (Architecting -> Planning -> Executing).
- Monitors for Signals.
- Handles interruptions and resumption.

### 2. The Architect (`src/workflow/architect.ts`)
Responsible for high-level design.
- **Input**: User prompt, global constraints, evolution log.
- **Output**: `architecture.md`.
- **Role**: Defines *what* needs to be built and the overall strategy.

### 3. The Planner (`src/workflow/planner.ts`)
Responsible for tactical execution steps.
- **Input**: Architecture, agent capabilities, active signals, completed tasks.
- **Output**: `plan.yml`.
- **Role**: Breaks the architecture into a dependency graph of `Task`s. It assigns each task to a specific Agent.

### 4. The Executor (`src/workflow/executor.ts`)
Responsible for getting things done.
- **Input**: `plan.yml`.
- **Action**: Executes tasks in topological order (respecting dependencies). Parallelizes independent tasks.
- **Mechanism**: Uses `AgentRunner` to invoke the assigned agent for each task.
- **Signal Check**: After every task, it checks for new signals.

## Data Models & Files

The system relies on file-based state and communication, making it transparent and easy to debug. All state files are stored in `.nexical/`.

### 1. Execution State (`.nexical/state.yml`)
Tracks the current status of the session.
```yaml
session_id: "2023-10-27-10-00-00"
status: "EXECUTING" # or PLANNING, INTERRUPTED, etc.
loop_count: 1
tasks:
  completed: ["task-id-1", "task-id-2"]
  pending: ["task-id-3"]
last_signal: null
```

### 2. Signals (`.nexical/signals/`)
Agents communicate issues by writing files here.
- **Format**: Markdown or Text files.
- **Naming**: Often contains the signal type, e.g., `REPLAN_task-03.md`.
- **Content**: The reason for the signal.
- **Effect**: The Orchestrator pauses execution, reads the signal, and transitions state (e.g., back to PLANNING).

### 3. The Plan (`.nexical/plan.yml`)
The roadmap for the Executor.
```yaml
plan_name: "Implement Auth"
tasks:
  - id: "task-1"
    message: "Create user schema"
    agent: "coder"
    dependencies: []
  - id: "task-2"
    message: "Implement login API"
    agent: "coder"
    dependencies: ["task-1"]
```

### 4. Architecture (`.nexical/architecture.md`)
The high-level design document generated by the Architect. It serves as the "north star" for the Planner.

### 5. Evolution Log (`log.md`)
A persistent record of *why* changes happened. When a signal causes a re-plan, the reason is logged here. This prevents the system from making the same mistake twice.

## Workflow Lifecycle

### Phase 1: Initialization
The Orchestrator loads the `state.yml`. If none exists, it starts a new session. If one exists, it resumes (useful for crashing or stopping and starting later).

### Phase 2: Architecting
**State: ARCHITECTING**
The `Architect` agent is called. It reads the user's request and generates `architecture.md`.

### Phase 3: Planning
**State: PLANNING**
The `Planner` agent is called. It reads `architecture.md` and generates `plan.yml`.
*Note: If a REPLAN signal occurred, the Planner also reads the signal reason to adjust the new plan.*

### Phase 4: Execution
**State: EXECUTING**
The `Executor` iterates through the tasks in `plan.yml`.
1.  **Select Task**: Finds a task whose dependencies are met.
2.  **Run Agent**: Calls `AgentRunner` to execute the task.
3.  **Check Signal**: Checks `.nexical/signals/`.
    - **If Signal Found**:
        - Stop execution.
        - Log the signal to `log.md`.
        - Update State to `INTERRUPTED`.
        - Transition to `PLANNING` (for REPLAN) or `ARCHITECTING` (for REARCHITECT).
    - **If No Signal**: Mark task as completed and continue.

### Phase 5: Completion
**State: COMPLETED**
When all tasks are done, the system exits successfully.

## Agent Integration

### Agent Definitions (`AGENTS.md` / `agents/*.yml`)
Agents are defined by their capabilities and configuration.
- **Profile**: A YAML file (e.g., `coder.agent.yml`) defining the agent's name, description, and the command to run it.

### The CLI Bridge
The system uses a `CommandRegistry` and `AgentRegistry`. The default `cli` plugin allows you to integrate any external tool.
- **Command**: The executable to run (e.g., `python`, `node`, `my-cli-tool`).
- **Args**: Arguments to pass.
- **Prompting**: The system renders a prompt (using `PromptEngine` and templates) and passes it to the agent via arguments or stdin.

### Personas
Personas are markdown files in `.nexical/personas/`. They define a "character" or specific expertise (e.g., `security-expert.md`).
- **Usage**: When a task specifies a persona, the `AgentRunner` injects the persona's content into the prompt sent to the agent.
- **Benefit**: Allows the same generic "coder" agent to behave differently when working on security vs. UI tasks.

## Bridging Systems & Resumability

### How to Bridge CLI Tools
To add a new tool (e.g., a Rust-based code analyzer):
1.  Create an agent profile `analyzer.agent.yml`.
2.  Set the command to your binary path.
3.  Define the arguments to accept the prompt/input.
The Orchestrator will now treat your Rust binary as a first-class agent, capable of receiving tasks and even generating signals (if your tool writes to the `signals/` directory).

### Resuming Workflows
Because state is persisted to `state.yml` after every major step:
1.  **Crash Recovery**: If the process is killed, running it again reloads `state.yml`. It sees `status: EXECUTING` (or `INTERRUPTED`) and picks up exactly where it left off.
2.  **Human Intervention**: A user can pause the system, manually edit `plan.yml` or `architecture.md`, and then restart the system. The Orchestrator will use the updated files.

## Summary
This execution model provides a robust, self-correcting framework for autonomous development. By decoupling the **Controller** (Orchestrator) from the **Workers** (Agents) via file-based signals and state, it achieves high flexibility, observability, and resilience.
