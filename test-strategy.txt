This is a complex, distributed system architecture involving a CLI entry point, structured data definitions (YAML), potentially non-deterministic planning (assuming an LLM back-end for the "planner"), graph execution logic, and external system interactions (agents).

Testing such a system requires a layered strategy that moves from isolating individual components to validating the entire workflow. Because parts of this system (the planner) are likely non-deterministic, and other parts (the agents) create side effects, reliance on mocking and snapshots will be heavy.

Here is a comprehensive testing strategy for your CLI orchestration system.

---

### 1. The Testing Pyramid Strategy

We will organize tests into three main tiers based on scope and execution speed.

#### Tier 1: Unit Tests (Fast, Isolated, Deterministic)
*Goal: Validate the smallest testable parts of the core logic in isolation, without external dependencies (no file system access, no network calls).*

* **YAML Schema Validation:**
    * Write tests that define valid and invalid `capabilities.yml` and agent definition structs.
    * Assert that your parsers accept valid configs and throw specific, descriptive errors on invalid ones (missing keys, wrong types).
* **DAG Logic (The Executor Core):**
    * Test the graph data structure independently.
    * Create graph instances with known dependencies. Test topological sorting algorithms to ensure tasks are ordered correctly.
    * Test cycle detection: ensure the system rejects a plan if A depends on B and B depends on A.
* **Prompt Interpolation Engine:**
    * Test the function responsible for taking an agent's template and variables and generating the final prompt/command.
    * *Input:* Template string, variable map. *Output:* Interpolated string.
* **CLI Argument Parsing:**
    * Test the entry point functions. Given a set of command-line arguments (flags, prompt text), assert that the internal configuration object is populated correctly.

#### Tier 2: Integration Tests (Medium Speed, Mocked Boundaries)
*Goal: Test how components interact. Crucially, this is where you handle the non-determinism of the planner and the side effects of agents.*

* **Mocking the Planner (Handling Non-Determinism):**
    * If your planner uses an LLM, you cannot rely on live calls for automated testing.
    * Create a mock planner interface. For a specific input prompt and capabilities list, return a *hardcoded, pre-defined DAG structure*.
    * This allows you to test that *if* the planner yields Plan X, the executor correctly performs steps A, B, and C in order.
* **Mocking the Agents (Avoiding Side Effects):**
    * You don't want tests actually running `rm -rf /` or calling OpenAI's image API.
    * Create mock Agent Runners. When the executor tells a "CLI agent" to run a command, the mock intercepts it, asserts the command string is correct, and returns a canned stdout/stderr response (success or failure).
* **Filesystem Integration (Using Temporary Directories):**
    * Test the logic that locates the project root versus `./dev_project`.
    * Test the "Save to History" functionality. Use test fixtures that create a temporary directory structure, run the save function, and assert the resulting JSON/YAML files exist and contain the correct plan data.

#### Tier 3: End-to-End (E2E) System Tests (Slow, Realistic)
*Goal: Test the fully assembled CLI binary in an environment resembling production. These tests prove it "actually works."*

* **The "Golden Master" / Snapshot Approach:**
    * Since the output is complex (stdout messages + history files + potential project file changes), use snapshot testing.
    * Set up a controlled environment with specific capabilities and deterministic mock agents (e.g., "echo" agents).
    * Run a specific CLI command.
    * Capture all outputs: the final stdout message and the contents of the generated `history/` directory.
    * On the first run, save these outputs as the "Golden Master". On subsequent runs, compare the new output exactly against the master. Any deviation is a failure.
* **Error Scenarios:**
    * Test running the CLI in a directory with no `capabilities.yml`.
    * Test running with a corrupted YAML definition.
    * Test a scenario where a mocked agent returns a failure exit code. Assert the DAG execution stops and the CLI reports the error correctly to the user.

### 2. Strategy for Project-Specific Agent Definitions

A critical requirement is supporting agents added by users in their specific projects, outside the core CLI codebase. If these agents are badly defined, they will break the orchestration.

We need a "Contract Testing" approach for these external definitions.

**The Solution: A Shared Validation Test Suite**

You should ship a testing utility (perhaps as a subcommand of your main CLI, e.g., `your-cli validate-agents`) that project owners can run in their own CI pipelines.

1.  **Define the Schema Contract:** Create a rigorous JSON Schema or CUE definition for what constitutes a valid agent YAML file.
2.  **The Validation Tool:** This tool will scan the project directory for agent definitions.
    * **Static Analysis:** It validates every found YAML against the Schema Contract. If a required field like `image_provider` is missing for an image-gen agent, it fails instantly.
    * **Dry-Run / Interpolation Test:** The tool should attempt to interpolate the agent's prompt template using placeholder variables. If the template references variables not defined in the args, it fails.
    * **Mock Execution (Optional but recommended):** For CLI-type agents, the validator could attempt to execute the command in a sanitized shell environment appended with a `--help` or `--version` flag just to ensure the underlying executable exists and is runnable (dangerous if not sandboxed properly, maybe skip for V1).

**Workflow for a Project Developer:**
When a developer adds a new specialized agent to their project:
1. They create `my_new_agent.yml`.
2. They run `your-cli validate-agents ./my_project_dir`.
3. The tool reports: "✅ my_new_agent.yml passes schema validation." or "❌ Error: missing 'command' key."

### 3. Summary of Test Implementation Steps

| Test Type | Scope | Tooling Examples (Python) | Tooling Examples (Go) | Key Strategy |
| :--- | :--- | :--- | :--- | :--- |
| **Unit** | YAML parsers, DAG graph logic, Template engine | `pytest` | `testing` package, `testify/assert` | Strict input/output validation. |
| **Integration** | Planner <-> Executor, Executor <-> Agent Runner | `pytest`, `unittest.mock` | `gomock` or interface mocking | **MOCK THE PLANNER.** Mock the actual agent execution. Use temp dirs. |
| **E2E** | The compiled CLI binary | `pytest-console-scripts`, custom shell scripts | `testscript` (highly recommended for CLI tools) | Snapshot outputs. Test happy path and failure paths. |
| **External Agents** | User-provided YAML definitions | A custom `validate` subcommand shipped with your CLI | Same | Static schema validation of user files against a central contract. |

### Key Takeaway for Success

The most critical part of this strategy is **isolating the non-deterministic planner from the deterministic executor**.

If you try to test the executor by making real calls to an LLM-based planner, your tests will be flaky and fail randomly. You must have a test seam where you can inject a pre-calculated DAG so you can rigorously test the execution engine's ability to handle dependencies and perform tasks.
