# Integration & E2E Test Cases

This document outlines the hyper-comprehensive testing suite for the `@nexical/engine`. It covers happy paths, complex agent orchestrations, failure recovery, and esoteric edge cases.

---

## 1. Core Workflow & Transitions

### CASE-1.1: Standard New Feature Request (Happy Path)
*   **Goal**: Full lifecycle verification.
*   **Assertions**: `IDLE` -> `ARCHITECTING` -> `PLANNING` -> `EXECUTING` -> `COMPLETED`. Verified merged commits.

### CASE-1.2: Immediate Re-Architecture
*   **Goal**: Test `REARCHITECT` signal during the Architect phase.
*   **Scenario**: User provides feedback that forces a design pivot before planning starts.
*   **Assertions**: Loop count increments, `Architecture.md` updates, transitions to `PLANNING` only after user approval.

### CASE-1.3: Empty/Degenerate Request
*   **Goal**: Handle prompts that result in "No action needed".
*   **Assertions**: Planner produces 0 tasks; system moves to `COMPLETED` gracefully without error.

### CASE-1.4: Invalid Plan Schema Reception
*   **Goal**: Resilience against malformed LLM YAML output.
*   **Assertions**: `Plan.fromYaml` throws Zod error; Workflow catches it and retries Planning with error feedback.

## 2. Advanced Agent Orchestration

### CASE-2.1: Parallel Execution & Merge Ordering
*   **Goal**: Ensure Task B doesn't merge before Task A if they are in different layers.
*   **Assertions**: Layer order is strictly enforced by the Executor loop.

### CASE-2.2: Circular Dependency Detection
*   **Goal**: Ensure the `PlanGraphValidator` catches cycles generated by the LLM.
*   **Assertions**: System refuses to enter `EXECUTING` state and emits a failure signal.

### CASE-2.3: Architect Persona Injection
*   **Goal**: Verify the Architect correctly creates persona files in `.ai/personas/`.
*   **Assertions**: Verify side-effect files exist after Architecting phase.

### CASE-2.4: Ambiguous Task Assignment
*   **Goal**: Planner assigns a task to a non-existent skill.
*   **Assertions**: Executor fails early with "Skill not found" error; transition to `FAILED`.

## 3. Signals & Multi-Turn Intelligence

### CASE-3.1: Re-Planning Loop with Evolution Log
*   **Goal**: Verify that a structural failure in Execution triggers a Re-Plan that *uses* the failure feedback.
*   **Assertions**: Turn 2 Planner prompt includes the error message from Turn 1.

### CASE-3.2: Re-Architecture Triggered from Execution
*   **Goal**: Deep failure where even the design is flawed.
*   **Assertions**: Signal `REARCHITECT` bubbled from Execution results in `ARCHITECTING` state entry.

### CASE-3.3: Interactive Feedback during Execution
*   **Goal**: User injects feedback mid-execution via the `RuntimeHost.ask`.
*   **Assertions**: Execution pauses, state reflects input, and logic adjusts if signal emitted.

### CASE-3.4: Max Turn Limit Circuit Breaker
*   **Goal**: Prevent infinite billing/loops.
*   **Assertions**: System terminates after `maxLoops` with `FAILED` status.

## 4. Inter-Agent Communication (The Oracle Bus)

### CASE-4.1: Cross-Agent Clarification (Full Loop)
*   **Goal**: Task asks Architect (via Bus), Architect asks User, Answer flows back.
*   **Assertions**: Verify `req_*.json` and `res_*.json` contents and timestamps.

### CASE-4.2: Bus Timeout / Message Drop
*   **Goal**: Testing the `waitForResponse` timeout logic.
*   **Assertions**: Task fails or retries if response doesn't appear in time.

### CASE-4.3: Concurrent Requests Collision
*   **Goal**: Two parallel tasks ask the Oracle at the same time.
*   **Assertions**: Correlation IDs ensure Answer A goes to Task A and Answer B to Task B.

### CASE-4.4: Corrupt IPC Payload
*   **Goal**: Inject malformed JSON into the Inbox.
*   **Assertions**: Oracle logs a warning but continues watching; does not crash.

## 5. Isolation & GIT Transactions

### CASE-5.1: Worktree Leak Prevention
*   **Goal**: Verify `.worktrees/` are cleaned up even if a task crashes with a hard exception.
*   **Assertions**: `git worktree list` shows 0 entries after Orchestrator exits.

### CASE-5.2: Merge Conflict Resolution
*   **Goal**: Two tasks change the same line.
*   **Assertions**: Orchestrator emits a specific `MERGE_CONFLICT` error; does not clobber the active branch.

### CASE-5.3: Sparse Checkout Integrity
*   **Goal**: Verify task only sees files defined in its skill's `sparse_checkout` list.
*   **Assertions**: Task attempts to read outside the set; verify `fs` access denied or file missing.

### CASE-5.4: Submodule Sync Failure
*   **Goal**: Skill requires `submoduleUpdate`, but network/repo is down.
*   **Assertions**: Task fails gracefully; merge is not attempted.

### CASE-5.5: Stale Worktree Pruning
*   **Goal**: Startup recovery of previous crashes.
*   **Assertions**: Manually created stale worktrees are removed by `Orchestrator.init()`.

## 6. Resilience & State Persistence

### CASE-6.1: State Resumption (Warm Boot)
*   **Goal**: Kill process after 50% completion. Resume.
*   **Assertions**: Skips finished tasks, confirms `EngineState.loop_count` is preserved.

### CASE-6.2: State File Lockdown (Concurrency)
*   **Goal**: Prevent two Orchestrators from running in the same .ai directory.
*   **Assertions**: Second instance throws "Workspace locked" error.

### CASE-6.3: Partial Write Recovery
*   **Goal**: State file (`state.yml`) is corrupted/truncated mid-write.
*   **Assertions**: System falls back to a backup `.bak` OR enters `FAILED` safely rather than crashing.

### CASE-6.4: External Interrupt Signal
*   **Goal**: Dropping a `SIGNAL_STOP` file in `.ai/signals`.
*   **Assertions**: Executor terminates current task layer safely and exits.

## 7. Skill Pipeline Lifecycle

### CASE-7.1: Pre-Analysis Environment Validation
*   **Goal**: Check if `pre_analysis_commands` can stop execution if env is bad (e.g., `command -v node`).
*   **Assertions**: Skill fails before LLM call if pre-analysis fails.

### CASE-7.2: Post-Execution Formatting
*   **Goal**: Verify `post_execution_commands` (like Prettier) run inside the worktree before merge.
*   **Assertions**: Files in main branch are formatted correctly.

### CASE-7.3: Multi-Phase Verification Retry
*   **Goal**: Verification Driver detects error -> Execution retries -> Success.
*   **Assertions**: Trial 1: FAIL, Trial 2: PASS.

### CASE-7.4: Verification Driver Timeout
*   **Goal**: Verification phase hangs.
*   **Assertions**: Skill fails or times out without blocking the entire Orchestrator indefinitely.

## 8. Esoteric & Security Edge Cases

### CASE-8.1: Directory Traversal via Prompt
*   **Goal**: LLM attempts to write to `../../etc/passwd` or outside workspace.
*   **Assertions**: `FileSystemService` or Git isolation prevents writes outside the sandbox.

### CASE-8.2: Environment Variable Leakage
*   **Goal**: Verify sensitive host ENV vars (like AWS keys) are NOT leaked into the Task's shell context unless explicitly mapped.
*   **Assertions**: Task shell cannot see host-specific private keys.

### CASE-8.3: Binary Artifact Handling
*   **Goal**: Task generates a `.png` or `.exe` file.
*   **Assertions**: Git merges binary files correctly; state handles non-string artifacts.

### CASE-8.4: Path Case-Sensitivity Issues (OS-specific)
*   **Goal**: Verify filename collisions on macOS/Windows vs Linux (e.g. `File.ts` vs `file.ts`).
*   **Assertions**: System handles or validates path casing consistently.

### CASE-8.5: Large Prompt Overflow
*   **Goal**: Extremely large architecture or plan exceeds Driver/LLM context limits.
*   **Assertions**: System handles truncation or emits a "Context Exceeded" failure gracefully.

## 9. Advanced Communication Failure Modes

### CASE-9.1: Inbox Replay on Crash
*   **Goal**: Verify reliability of the `FileSystemBus` when the process crashes mid-processing.
*   **Steps**:
    1.  Write `req_A.json` to inbox.
    2.  Crash `ArchitectAgent` before it deletes the file.
    3.  Restart Agent.
*   **Assertions**: Agent picks up `req_A.json` again (at-least-once delivery) or ignores if strictly processed (idempotency).

### CASE-9.2: Outbox Orphan Cleanup
*   **Goal**: Ensure `res_*.json` files don't accumulate forever if the requestor dies.
*   **Assertions**: Start a task that requests clarification, then kill the task. The `res_` file should eventually be cleaned up by a separate GC process or manually.

### CASE-9.3: Massive Payload Handling
*   **Goal**: Stress test `fs.readFile` / `JSON.parse` with a 10MB message.
*   **Assertions**: Bus handles large files without OOM crash; Agent refuses or processes slowly but correctly.

### CASE-9.4: Correlation ID Path Traversal
*   **Goal**: Malicious actor injects `correlationId: "../../etc/passwd"`.
*   **Assertions**: `FileSystemBus` validates filenames to prevent writing outside `.ai/comms`.

### CASE-9.5: Zero-Length Message File
*   **Goal**: File watcher triggers on empty file creation (before write completes).
*   **Assertions**: `JSON.parse` fails; Bus retries or ignores until content exists.

## 10. Git & Worktree Edge Cases

### CASE-10.1: Worktree Limit Exhaustion
*   **Goal**: System behavior when `max_worktrees` config is exceeded.
*   **Steps**: Lower limit to 2. Schedule 5 parallel tasks.
*   **Assertions**: Executor throttles creation, queuing tasks 3-5 until 1-2 finish.

### CASE-10.2: Dirty Main Branch
*   **Goal**: User has uncommitted changes in `src/`.
*   **Assertions**: `Executor` refuses to start strategies that might rely on clean HEAD, OR creates worktree from HEAD successfully ignoring dirty state (correct behavior for worktrees).

### CASE-10.3: Detached HEAD State
*   **Goal**: Repo is in detached HEAD state (e.g. checking out a specific tag).
*   **Assertions**: Executor can still manage branches or fails gracefully with "Please checkout a branch" message.

### CASE-10.4: Submodule Auth Failure
*   **Goal**: Project contains a private submodule the Agent cannot access.
*   **Assertions**: `git submodule update` fails. Task fails. Main repo remains consistent.

### CASE-10.5: Binary Merge Conflicts
*   **Goal**: Two tasks generate valid but conflicting binary assets (e.g. `icon.png`).
*   **Assertions**: Git auto-merge fails. Executor reports conflict requiring manual intervention.

### CASE-10.6: File Locking (Windows Simulation)
*   **Goal**: Simulate `EBUSY` or `EPERM` when trying to delete a worktree.
*   **Assertions**: `GitService` retries deletion or logs warning/leaves garbage without crashing the orchestrator.

## 16. Git Extreme & File System Oddities

### CASE-16.1: Git Hooks Interference
*   **Goal**: User has a `pre-commit` hook that fails or modifies files.
*   **Assertions**: Executor detects hook failure and reports it clearly, or handles auto-formatted changes.

### CASE-16.2: Git LFS Pointer Handling
*   **Goal**: Task interacts with Large File Storage (LFS) assets.
*   **Assertions**: `GitService` correctly pulls actual files, not just pointers (requires `git lfs pull` support or config check).

### CASE-16.3: Max Path Length (Windows Simulation)
*   **Goal**: Task generates deeply nested directory structures (>260 chars).
*   **Assertions**: System handles it or fails gracefully with a clear "Path too long" error.

### CASE-16.4: Branch Name Collisions
*   **Goal**: Task ID generates a branch name that conflicts with an existing directory or ref.
*   **Assertions**: `GitService` sanitizes branch names or retries with a suffix.

### CASE-16.5: Corrupt Git Index
*   **Goal**: `.git/index.lock` exists or index is corrupt.
*   **Assertions**: System attempts to clear stale locks or fails with a "Git Repair Needed" message.

### CASE-16.6: Symlink Attacks
*   **Goal**: Task attempts to create a symlink pointing to `/etc/passwd`.
*   **Assertions**: Security check prevents creation or Git config `core.symlinks=false` is respected.

### CASE-16.7: Read-Only Filesystem Mount
*   **Goal**: Run Engine on a read-only volume.
*   **Assertions**: Immediate failure during `init` (cannot write `.ai` logs) with clear permission error.

### CASE-18.4: Unicode & Emoji Support

## 19. Chaos Engineering & Interrupts

### CASE-19.1: Network Partition (Drivers)
*   **Goal**: Pull network cable (disable wifi) during `driver.execute`.
*   **Assertions**: Driver throws Timeout/Network Error; Task Retry Logic kicks in.

### CASE-19.2: Sigkill (Hard Crash)
*   **Goal**: `kill -9` the process.
*   **Assertions**: On restart, `GitService` detects stranded worktrees and `loadState` sees `EXECUTING` status but no running process -> Auto-Resumes or prompts user.

### CASE-19.3: Disk Full
*   **Goal**: Simulate `ENOSPC` when writing `state.yml`.
*   **Assertions**: Atomic write fails; previous state file preserved. System halts with "Log Fatal".

### CASE-19.4: Memory Limit (OOM)
*   **Goal**: Task or Prompt Engine consumes 4GB+ RAM.
*   **Assertions**: Node process crashes. Supervisor (if any) or User restarts. State file was updated *before* the memory spike (Checkpointing).

## 20. Skill Definition Edge Cases

### CASE-20.1: Missing Dependency
*   **Goal**: Skill requires `jq` but it's not installed.
*   **Assertions**: `pre_analysis` check using `command -v jq` fails the skill fast.

### CASE-20.2: Circular Skill Dependencies
*   **Goal**: Skill A requires Skill B; Skill B requires Skill A.
*   **Assertions**: Registry load or runtime resolution validation loop handling.

### CASE-20.3: Shell Injection in Params
*   **Goal**: `params` contains `; rm -rf /`.
*   **Assertions**: `PromptEngine` renders text safely; `ShellService` uses `spawn` (exec array) not `exec` (shell string) to prevent execution.

### CASE-20.4: Timeout Enforced
*   **Goal**: Skill execution runs forever (e.g. `sleep 10000`).
*   **Assertions**: `Executor` kills child process after configured `task_timeout`.
